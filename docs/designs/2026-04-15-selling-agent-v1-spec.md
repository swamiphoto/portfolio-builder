## V1 Wedge

Target user:

- landscape and fine-art photographers
- already have a body of finished work
- want to turn existing photos into sellable listings without repetitive admin

Core promise:

"Turn 20 finished photos into ready-to-sell listings in under 10 minutes."

This is the whole game. Not "career autopilot." Not "submit everywhere." Just reduce the ugly repetitive work.

## Why This Wedge

- it has direct ROI
- it uses the canonical library this product already needs
- it is more structured than publication pitching
- it produces obvious success metrics
- it can expand into store support later instead of fighting the roadmap

## V1 Scope

### User flow

1. User imports or uploads photos into the library.
2. User marks selected assets as `sellable`.
3. User fills or accepts generated metadata:
   - title
   - caption
   - keywords
   - category
   - orientation
   - release status
   - pricing preset
4. User connects 2 destinations.
5. Agent prepares destination-specific listing packages.
6. User reviews a queue of prepared listings.
7. User clicks approve.
8. System submits where integration exists, otherwise exports a ready package.
9. User sees listing status and history.

### What V1 includes

- sellable asset flag
- asset metadata panel
- pricing presets
- rights/release flags
- 2 destination adapters
- approval queue
- destination-specific file prep
- submission/export history

### What V1 does not include

- publication submissions
- contest submissions
- dynamic pricing optimization
- inbox parsing
- taste-learning loop
- browser automation against hostile platforms
- auto-submission without approval

## Best First Destinations

V1 should start with:

### Destination A: Print-ready export

This can be framed as a print sales destination or fulfillment prep path.

Why first:

- lowest platform risk
- immediate user value
- proves metadata + pricing + variant generation
- does not depend on a partner API on day one

What it does:

- generate print-ready files and web preview files
- attach pricing preset
- attach title/caption/keywords
- export a package for a supported sales channel or internal storefront flow

### Destination B: One stock/licensing channel

Why second:

- clearer marketplace story
- validates channel-specific metadata mapping
- stronger commercial signal than export-only

Selection rule:

- choose the destination with the cleanest ingestion path, not the biggest brand

## Product Surface

### 1. Library additions

Each asset gets:

- `sellable: boolean`
- `title`
- `caption`
- `keywords: string[]`
- `category`
- `orientation`
- `releaseStatus: "none" | "property" | "model" | "both"`
- `licenseAvailability: "editorial" | "commercial" | "both"`
- `pricingPresetId`
- `destinationEligibility: string[]`
- `submissionHistory: []`

This is the dependency. Without this, the agent is just guessing.

### 2. Pricing presets

User defines 2-4 presets, for example:

- Small Print
- Large Print
- Digital License

Each preset stores:

- label
- price
- currency
- usage notes

V1 does not need algorithmic pricing. Fixed presets are enough.

### 3. Destination model

Each destination stores:

- `id`
- `name`
- `type`
- `integrationMode: "api" | "export"`
- required metadata fields
- image format rules
- min/max dimensions
- allowed license types
- status

### 4. Approval queue

A new admin view:

- list of prepared listings
- one row per asset x destination
- validation errors inline
- preview of mapped title/caption/keywords/pricing
- `Approve`
- `Needs edits`
- `Skip`

### 5. Submission history

Per asset, show:

- where it was sent
- when
- what metadata was used
- current status
- error if rejected by validation

## Recommended UI

Add a new area in admin:

- `Library`
- `Pages`
- `Selling`

Inside `Selling`:

1. `Queue`
2. `Destinations`
3. `Pricing`
4. `History`

Keep it boring. This is operations software.

## Technical Shape for This Repo

This repo already has:

- library and asset handling
- admin area
- config-backed persistence
- site/store-adjacent roadmap

So V1 should be implemented as:

- library metadata extensions in the same config model
- selling config under per-user storage
- destination adapter layer in server-side utilities
- new admin pages/components for queue, pricing, history

## Success Metrics

Primary:

- user prepares 20 listings in under 10 minutes
- at least 80% of prepared queue items need no manual metadata edits

Secondary:

- user connects 2 destinations
- user approves at least 10 listings in first session
- less than 10% of queue items fail destination validation

## Risks

### Metadata quality

If captions and keywords are bad, the agent feels fake.

### Rights mistakes

If commercial/editorial eligibility is wrong, trust is gone.

### Wrong first destination

If the first channel has bad ingestion ergonomics, the product looks broken even if the core idea is right.

## Build Order

1. Extend library asset model with selling metadata.
2. Add pricing preset config.
3. Add destination model and one export adapter.
4. Add selling queue UI.
5. Add submission history.
6. Add one true marketplace adapter.

## Recommendation

Build this before publication workflows.

Publication tooling is attractive, but selling is the cleaner wedge:

- more structured
- more measurable
- more adjacent to the current product
- less dependent on subjective editorial fit

After this works, add publication submissions as a second module using the same asset metadata and approval queue.
