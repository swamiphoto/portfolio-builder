# Design: Selling Agent for Photographers

## Problem Statement

Photographers do not just need a portfolio site. They also need distribution.

Today that work is fragmented and annoying:

- upload to a portfolio site
- upload again to stock/marketplace platforms
- rewrite captions, tags, and pricing over and over
- hunt for publication calls for entry
- resize, rename, and reformat assets for each destination
- remember what was submitted where, when, and with what rights

The idea is a "selling agent" that acts like a personal submissions assistant. A photographer gives the system permission to use selected photos, and the system helps place them into marketplaces and publication opportunities.

## Hard Truth

This is feasible to build.

It is not feasible to build well as a generic "submit everywhere automatically" button in v1.

Why:

- many destinations do not have stable write APIs
- submission rules differ a lot by destination
- rights, exclusivity, embargo, and licensing terms can conflict
- publication submissions are editorial decisions, not just form filling
- some platforms will classify headless automation as abuse
- photographers care a lot about brand fit, not just volume of submissions

So the right product is not "spray every image everywhere."

The right product is "agentic distribution with human approval and destination-specific adapters."

## Strongest Version of the User Need

The actual user need is probably:

"I want one place to manage my photos, metadata, rights, and availability, then have an agent find good-fit opportunities, prepare submissions, and reduce the repetitive work to one-click approval."

That is much better than:

"Bot uploads my work all over the internet."

The first one is premium workflow software.
The second one gets you banned and produces junk submissions.

## Premises

1. Photographers want help with distribution, not just site creation.
2. The painful part is repetitive admin plus opportunity discovery, not raw file upload alone.
3. Human approval before submission is a feature, not friction.
4. Metadata, rights, and usage constraints must be first-class data in the library.
5. Destination coverage will need curated adapters, not one universal integration layer.

## Feasibility Breakdown

### Technically feasible now

- maintain a canonical asset library with metadata
- store rights state per image
- generate resized variants per destination
- prefill submission packages
- track submission history and status
- match photos to opportunities using rules or AI ranking
- submit automatically to destinations with real APIs or stable intake formats
- send drafts for human approval everywhere else

### Hard but feasible later

- marketplace-specific pricing suggestions
- brand-fit scoring per publication
- dedupe and conflict detection across destinations
- license conflict prevention
- inbox parsing for acceptance/rejection tracking
- "best next place to send this photo" recommendations

### Dangerous / weak v1 bets

- fully automatic submission to arbitrary publications
- browser automation against platforms with anti-bot protections
- automatic rights decisions without explicit user policy
- pushing the same image to conflicting stock platforms

## What This Means for This Repo

This repo already wants to be the photographer's system of record:

- import from existing photo sources
- canonical library
- page and gallery composition
- future store support

That makes this feature adjacent, not random.

But it is a second product layer. It needs:

- richer asset metadata
- rights and licensing model
- opportunity database
- adapter framework for destinations
- approval queue
- submission tracking

The current platform direction supports this. The library is the key dependency.

## Approach Options

### Approach A: Submission Assistant

Summary: The product finds opportunities, prepares assets and copy, and gives the photographer an approval queue. The user clicks approve, then the system submits where supported or opens the destination with everything prepared.

Effort: M
Risk: Low

Pros:
- useful fast
- avoids compliance traps
- fits the current portfolio/library product
- can start with publications before deep marketplace integrations

Cons:
- less magical than full autopilot
- user still stays in the loop
- some destinations will remain semi-manual

### Approach B: Marketplace Autopilot

Summary: Start only with 2-3 sales channels that have stable APIs or predictable workflows, for example stock marketplaces or print destinations, and automate those end to end.

Effort: L
Risk: Medium

Pros:
- clearer ROI
- revenue path is more obvious
- tighter adapter scope

Cons:
- narrower story
- publications side is deferred
- depends heavily on partner/API quality

### Approach C: Full Career Agent

Summary: One agent that discovers calls for entry, ranks images, rewrites captions, negotiates formats, submits automatically, tracks responses, and learns the photographer's taste over time.

Effort: XL
Risk: High

Pros:
- huge vision
- very differentiated if it works
- aligns with the "career autopilot" direction

Cons:
- too many moving parts for v1
- legal and platform-risk heavy
- hard to verify quality
- easy to become a demo instead of a real product

## Recommendation

Choose **Approach A**, then extend into **Approach B**.

That means:

1. Start with an approval-based submission queue.
2. Add 3-5 curated destinations only.
3. Make publications mostly "agent prepares, human approves."
4. Make marketplaces/print channels the first true autopilot integrations.

This preserves the useful part and avoids the fake-magic part.

## Narrowest Wedge

The narrow wedge is not "all photographers."

It is:

"landscape and fine-art photographers who already maintain a portfolio and want help turning finished work into submissions and sales without repetitive admin."

The first paid promise could be:

"Select 20 photos, connect 2 channels, get a weekly queue of matched opportunities and one-click prepared submissions."

That is concrete.

## Product Shape

### New data the platform would need

Per asset:

- title
- caption
- keywords
- location
- orientation
- release status
- licensing availability
- exclusivity flags
- pricing bands
- submission history
- destination eligibility

Per destination:

- type: marketplace, publication, print, contest
- required formats
- min/max dimensions
- rights constraints
- categories
- submission method
- status of integration: api, assisted, manual

### Key workflows

1. Photographer marks assets as sellable/submittable.
2. Agent scores opportunities against asset metadata and taste.
3. Agent creates a queue: "3 strong fits this week."
4. User reviews, edits, approves.
5. System submits or assists.
6. Submission statuses feed back into the asset history.

## Main Risks

### Product risk

If users do not trust the agent's taste, they will not let it submit.

### Legal/platform risk

Some destinations will not want automation. You need a policy per destination.

### Quality risk

Generic AI-generated captions and pitches will feel fake fast. Bad submissions hurt the photographer's reputation.

### Scope risk

Combining portfolio builder, import engine, store, marketplace distribution, and publication CRM in one jump is too much.

## What I Would Build First

V1:

- asset metadata model in the library
- rights/licensing flags
- "opportunities" table
- weekly recommendation queue
- submission package generation
- approval workflow
- 1 marketplace adapter
- 1 publication or call-for-entry adapter
- submission history UI

V1 success metric:

"A photographer submits 5 photos to 2 relevant destinations in under 10 minutes, with less than 2 minutes of manual editing."

## Verdict

Yes, feasible.

But only if you frame it as a **curated approval-first distribution agent**.

If you frame it as **submit my photos everywhere automatically**, it is a trap.

The product opportunity is real. The naive implementation is not.
