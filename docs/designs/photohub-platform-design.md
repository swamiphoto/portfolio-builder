## Problem Statement

Photographers want portfolio websites but are stopped by friction, not by lack of desire. The existing tools (Squarespace, Format, Pixieset, SmugMug, Zenfolio) all require the photographer to become a designer. They have to choose templates, configure layouts, drag widgets, manage settings. Most photographers have great taste but not design skills. They can tell good from bad, they just can't get there without friction.

The result: most photographers stay on Instagram. They never build a real portfolio site. The ones who do use SmugMug (which has "horrible" UX, per the builder) or Pixieset (which "feels more of a website builder than a photographer's dream portfolio site").

The core insight: **kill the friction between "I have photos" and "I have a beautiful site."**

## What Makes This Cool

The magic moment: paste your Instagram or portfolio URL, and get a beautiful, published site in under 2 minutes. No template picking, no layout configuration, no design decisions. Photos in, site out.

Then: a dead-simple block-based editor where photographers can rearrange photos, add text, switch themes, and manage pages through a sidebar-based interface that mirrors how they already think about their work (Cover, Galleries, About, Contact).

The platform starts from the photographer's existing work and builds the site for them. Every other tool starts from a blank template and asks the photographer to fill it in. That's the inversion.

## Constraints

- Built on top of an existing Next.js (pages router) + Tailwind + GCS codebase that already has: block-based gallery builder, photo library, slideshow admin, draft/publish workflow
- Single-developer project (AI-assisted)
- No existing auth, multi-tenancy, or user management
- GCS (Google Cloud Storage) for all image storage
- Current site is statically-oriented with ISR; platform version needs dynamic per-user routing

## Premises

1. **Friction is the barrier, not desire.** Photographers in the landscape community WANT sites but SmugMug is too painful and Instagram is "good enough."
2. **The magic moment is "paste URL, get site in 2 min."** That's acquisition. The page builder is retention.
3. **Block-based builder covers 80%+ of needs.** Existing block types (photo, stacked, masonry, text, video) plus themes provide sufficient creative control. No need for arbitrary CSS/layout power.
4. **Multi-tenancy can layer onto the existing codebase** without a full rewrite. Per-user GCS paths, auth, and routing can be added incrementally.
5. **Scraping/importing photos from Instagram, Squarespace, SmugMug, and any URL** is legally and technically viable as a core product feature.

## Approaches Considered

### Approach A: "Magic Import" MVP
Summary: Build ONLY the import-to-published-site flow. No page builder, no block editor, no store.
Effort: M (human: ~2 weeks / CC: ~2-3 hours)
Risk: Low. Pros: Tests core thesis fast. Cons: No editing after generation.

### Approach B: "Import + Edit"
Summary: Magic Import + existing block-based gallery builder for editing. Page-level sidebar. No store.
Effort: L (human: ~6 weeks / CC: ~6-8 hours)
Risk: Medium. Pros: Full import-edit-publish loop. Cons: Multi-tenancy complexity.

### Approach C: "Platform from Day 1" (CHOSEN)
Summary: Full vision. Import, page-level sidebar, gallery pages, single pages, library with caption sync, themes, store-ready architecture.
Effort: XL (human: ~4 months / CC: ~2-3 weeks)
Risk: High. Pros: Complete product vision. Cons: Months before anyone sees it.

## Recommended Approach

**Approach C: Platform from Day 1**, chosen by the builder. The existing codebase already has most building blocks. With AI-assisted development, XL effort compresses significantly.

### Page Types

**Cover Page** - Always first. Hero image, site name, tagline, optional featured galleries carousel, testimonials. Block-based.

**Gallery Page** - Thumbnails linking to sub-pages (albums). Layout options (2-col, 3-col, masonry, single). Albums are block-based pages reusing GalleryBuilder. Appears in nav.

**Single Page** - Block-based (photo, stacked, masonry, text, video). For About, Contact, Portfolio. Can include special blocks (carousel, testimonials). Appears in nav.

### Caption Syncing

Library is source of truth for captions. When a block references an image URL, the editor looks up the caption. When a caption is edited at the block level for a photo without one in the library, the library is updated. Next time that image is used anywhere, the caption auto-populates.

### Import / Onboarding Flow

1. Sign in with Google OAuth
2. "Your photos deserve a home." - paste a URL
3. Backend detects source type, uses appropriate strategy:
   - Instagram: gallery-dl or Instagram Graph API
   - SmugMug: gallery-dl (native support)
   - Squarespace: Playwright HTML scraping
   - Any URL: generic image scraper
4. Import progress in real-time
5. Auto-generation: score images, pick hero, group into collections, generate cover + gallery pages + about stub
6. Published immediately, user lands in editor

### Theme System

One theme initially (current site's visual language). Theme controls typography, colors, layout defaults, component styles. Implementation: CSS custom properties + theme config JSON. Theme dropdown in sidebar footer.

### Auth & Multi-Tenancy

- Google OAuth via NextAuth.js
- Per-user GCS paths: users/{userId}/photos/, users/{userId}/site-config.json
- Dynamic routing: photohub.com/u/[username] (v1), [username].photohub.com (v2)
- No billing in v1

### Nav Auto-Generation

Nav auto-generated from pages list. Pages with showInNav: true appear. Cover page is home link. Order matches sidebar order.

### Store (Future)

Any library image can be flagged forSale: true with pricing. "Buy" button appears wherever image is shown. Print fulfillment API integration (Printful, Prodigi). Deferred but data model accommodates it.

## Import Error States

| Scenario | User Sees | System Does |
|----------|-----------|-------------|
| URL unreachable / 404 | "We couldn't reach that URL. Check the link and try again." | Log error, no retry |
| gallery-dl fails (Instagram blocks) | "Instagram import isn't available right now. Try uploading photos manually." | Fall back to manual upload prompt |
| Too few images (<5) | "We found [N] photos. Want to import them anyway, or add more later?" | Allow proceed or manual upload |
| Import timeout (>120s) | "This is taking longer than expected. We'll email you when your site is ready." | Move to background queue, email on completion |
| Unsupported source | "We can't auto-import from this site yet. You can upload photos manually." | Log source for future support |

## Caption Sync Conflict Resolution

**Rule: last write wins.** If a caption is edited at the block level, it updates the library entry. If the library caption is edited directly, it does NOT retroactively update existing block-level captions (those are already committed). New references to that image will use the updated library caption. No warning on conflict. This is the simplest mental model.

## Auto-Generation Scoring Heuristic

```
HERO SELECTION:
  1. Prefer landscape orientation (aspect ratio > 1.3)
  2. Prefer highest resolution (width * height)
  3. Among top 5 candidates, prefer most recent
  4. If Instagram, prefer highest engagement (likes) if available

COLLECTION GROUPING:
  1. Group by date proximity (posts within 7 days = same collection)
  2. If fewer than 3 groups, split largest group by aspect ratio (landscapes vs portraits)
  3. Collection title = month + year ("December 2025") or first caption keyword
  4. Max 4 collections for auto-generation; remaining photos go to "More"

DEDUPLICATION:
  - Skip images with identical dimensions AND similar file size (within 10%)
  - Skip images under 400px on shortest side (likely thumbnails)
```

## Generic URL Scraper Behavior

Extracts all `<img>` tags with `src` attributes. Filters to images where shortest dimension > 600px (skip thumbnails, icons, avatars). Follows the first page only (no pagination). Deduplicates by URL. Max 100 images per import. If the page has lazy-loaded images, uses Playwright to scroll and trigger loading before extracting.

## Theme Dropdown State

When only one theme exists, the dropdown shows the theme name but is not interactive (no arrow, no options). When additional themes are added, the dropdown becomes active. The dropdown is always visible in the sidebar footer so users see that themes exist and are coming.

## Import Architecture (Vercel Constraints)

gallery-dl is a Python CLI tool and cannot run in Vercel serverless functions. The import engine runs as a **separate backend service**:

```
IMPORT ARCHITECTURE:
  Vercel (Next.js)                    Import Service (separate)
  +-----------------+                 +--------------------+
  | /api/import     | --- HTTP POST ->| /import            |
  | (enqueue job)   |                 | (gallery-dl/       |
  +-----------------+                 |  Playwright/        |
  | /api/import/    | <-- polling --- |  Instaloader)       |
  | status?id=xxx   |                 | Uploads to GCS     |
  +-----------------+                 +--------------------+
```

- Vercel API route creates an import job (stored in GCS or Redis)
- Import service polls for jobs, runs gallery-dl/Playwright, uploads images to user's GCS path
- Client polls `/api/import/status?id=xxx` every 3 seconds for progress
- Import service can run on: Railway, Fly.io, or a simple VPS
- No WebSocket needed; polling is simpler and works on Vercel

## Instagram Import Risk Mitigation

Instagram scraping is the **highest-risk dependency** in this product.

| Path | Stability | Effort | Trade-off |
|------|-----------|--------|-----------|
| gallery-dl | Low (Instagram blocks frequently) | Low (drop-in) | Fast to build, fragile |
| Instaloader | Medium (maintained, handles rate limits) | Low | Instagram-only, may break |
| Instagram Graph API | High (official, stable) | High (requires app review, user OAuth) | Stable but slow to set up, limited to user's own posts |
| Manual upload | High | Already built | No magic moment, but always works |

**V1 strategy:** Build with Instaloader as primary (better rate limit handling than gallery-dl). Build Instagram Graph API integration as stable fallback. Always offer manual upload as escape hatch. If Instagram import breaks, the product still works, it just loses the magic onboarding moment for Instagram users. SmugMug and Squarespace import via gallery-dl and Playwright respectively are more stable.

## ISR to Dynamic Rendering Migration

The existing site uses ISR (revalidate: 60) for gallery pages. The platform needs per-user dynamic rendering:

- **User sites (photohub.com/u/[username])**: Server-side rendered (SSR) via `getServerSideProps`. Reads user's `site-config.json` from GCS on each request. Cache with CDN (Vercel edge cache, `s-maxage=60, stale-while-revalidate`).
- **Admin/editor (/admin)**: Client-side rendered (CSR). No SSR needed. Auth check on mount.
- **Existing site (swamiphoto.com)**: Continues to use ISR. No changes needed. The platform is a separate Next.js app or a separate route group within the same app.
